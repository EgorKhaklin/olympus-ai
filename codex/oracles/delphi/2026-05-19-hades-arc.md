## Phase 0 — what's in plaintext today

```bash
$ cat state/config.json
{
  "llm": {
    "provider": "anthropic",
    "anthropic_api_key": "sk-ant-..."   ← plaintext, gitignored
  },
  ...
}
```

`state/config.json` is gitignored, but it sits in the operator's home directory at rest. A backup tool, a screenshare, a `cat` in a recorded terminal — any of these leaks the key. The macOS Keychain (and equivalents on Linux/Windows) gives us encryption-at-rest and access-on-prompt for ~zero added complexity.

## What ships

### `src/olympus/olympians/hades.py` — new Olympian

In mythology Hades held the riches of the underworld in his strongbox — keeper of what cannot be seen above ground. In Olympus, **Hades is the secrets vault**: deposit a secret, retrieve it back, forget it when done. He uses the `keyring` library which dispatches to:

- **macOS** → Keychain (encrypted, GUI-prompted)
- **Linux** → Secret Service (GNOME Keyring, KWallet)
- **Windows** → Credential Manager
- **Headless / no backend** → graceful fallback (returns `available=False`; callers use plaintext)

Public API:
```python
hades.deposit(key_name: str, secret: str) -> None
hades.retrieve(key_name: str) -> str | None
hades.forget(key_name: str) -> None
hades.where(key_name: str) -> Location   # "keychain" | "plaintext" | "env" | "unset"
hades.available() -> bool                # is keyring functional?
hades.backend_name() -> str              # e.g. "macOS.Keyring"
```

Service namespace: `"olympus"` (so all Olympus secrets group together in the OS UI).

### `runtime/config.py` patch

New: `effective_anthropic_api_key()` resolves in this order:
1. `ANTHROPIC_API_KEY` env var (always wins)
2. Hades (`hades.retrieve("anthropic_api_key")`)
3. `state/config.json::llm.anthropic_api_key` (legacy plaintext; emits one-line deprecation warning to stderr)
4. `None`

New: `migrate_plaintext_to_hades()` — reads `state/config.json::llm.anthropic_api_key`, if it looks like a real key (≥ 20 chars, starts with `sk-`), deposits to Hades, then **replaces** the field in config.json with the sentinel `"(in-hades)"`. Idempotent; safe to re-run.

### `runtime/llm_bridge.py::apply_to_environment` patch

If the env var is unset, the bridge calls `effective_anthropic_api_key()` and sets `ANTHROPIC_API_KEY` for the SDK. Lower-level code (the Anthropic SDK) is unchanged.

### `runtime/setup.py` patch

When the wizard's LLM step succeeds at testing the key, it now **deposits to Hades** instead of writing plaintext to config.json. The config gets `"anthropic_api_key": "(in-hades)"` as a marker.

### New CLI errand: `invoke vault`

- `invoke vault status` — what's stored, where, backend name
- `invoke vault deposit <name>` — prompts (no echo) for value, stores in Hades
- `invoke vault retrieve <name>` — prints whether the key is present (NOT the value)
- `invoke vault forget <name>` — removes from Hades
- `invoke vault migrate` — runs `migrate_plaintext_to_hades()` explicitly

### `doctor.py` check

New `_check_secrets()`:
- `✓ vault: 1 secret in hades (macOS.Keyring)` when keychain has it
- `! vault: anthropic_api_key in plaintext — run \`invoke vault migrate\`` when not
- `! vault: keyring unavailable on this platform; plaintext is the only option` when no backend

### `vault` added to Throne's `SAFE_ERRANDS`

So you can ask "is my key encrypted?" in chat.

## Constitution

| invariant | how Hades honors it |
|---|---|
| S1 | every deposit/forget recorded to Mnemosyne under `hades.event` — **value is NEVER logged** (length + hash only) |
| S3 (no surprise mutation) | migration only triggers explicitly via `invoke vault migrate` OR on first save through setup; reading never mutates |
| S6 | every "where is this key" answer is verifiable: `invoke vault status` shows the actual resolved location |
| S7 | constitutional actions stay CLI: Throne can READ vault status but cannot deposit/forget secrets |
| C7-equivalent | secrets storage is *configurable* (env / hades / plaintext); not hardcoded |
| AP1 | one Olympian ~150 LOC; no parallel secrets system; reuses `keyring` |
| AP7 (ledger-balancing) | migration is real — plaintext is *actually removed* from config.json, replaced with `"(in-hades)"` |

## Failure modes (named explicitly)

- **No keyring backend (Linux headless)**: `hades.available()` returns False; `effective_anthropic_api_key()` falls back to plaintext; doctor warns. Operator can install `python3-secretstorage` or run with env var.
- **OS prompts user for keychain access**: first-call may show a system dialog ("Allow Olympus to access the keychain"). Documented in QUICKSTART.
- **Operator revokes keychain access**: `hades.retrieve()` returns `None`; bridge falls back; doctor surfaces the issue.
- **Key rotated externally**: operator runs `invoke vault deposit anthropic_api_key`; old value overwritten.

---

## What does NOT ship this arc

- **No multi-key support per name** (one key per name; rotation = overwrite).
- **No remote secret management** (Vault by HashiCorp, AWS Secrets Manager, etc.). Out of scope.
- **No password-protected `state/config.json`** — non-secret fields stay plaintext.
- **No automatic re-deposit on key change** — operator explicitly runs `invoke vault deposit` when rotating.
